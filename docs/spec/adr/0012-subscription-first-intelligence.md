# ADR-0012: Subscription-first intelligence — MCP clients do the model work

Status: Accepted

## Context
User decision (2026-07-22): the model work Atlas needs (capture assistance, retrieval reasoning, proposal drafting, review synthesis) is already paid for once, at flat subscription rates, in the assistants the user already runs day to day — ChatGPT (Plus, via its custom-connector feature) and Codex CLI/other MCP-capable agents. Metering that same work again through Atlas's own OpenAI key (ADR-0011) means paying twice for the same tokens. This does not reopen ADR-0007 (MCP-first Hermes), ADR-0009 (direct entry capture), or ADR-0011 (OpenAI as provider) — it **extends** all three: Hermes was already MCP-first; this ADR generalizes *which* MCP clients count as Hermes and moves the generation work itself onto them, while the server keeps doing what only the server can do (store data, gate writes, compute embeddings for retrieval).

**Researched fact (2026-07-22):** ChatGPT's custom connectors (Developer mode, Plus and above) accept remote HTTPS MCP servers, but mandate **OAuth 2.1 with Dynamic Client Registration (RFC 7591)** for authorization — bearer tokens are not an accepted auth mode for this client. Codex CLI and other header-capable MCP agents have no such restriction; they can send `Authorization: Bearer <key>` directly. This asymmetry is the immediate driver for adding OAuth now rather than deferring it further (previously 06 §1 and 08 §6 treated OAuth as post-MVP, gated on "any third-party client" wanting it — that trigger has fired).

## Decision
- **Who does model work:** MCP clients connected to Atlas — the ChatGPT app (via its connector, OAuth) and Codex CLI or any other MCP-capable agent (via bearer key) — perform all reasoning: reading retrieved context, drafting proposals, answering questions, capturing entries in the user's words. This is Hermes (07), generalized from "Claude app / Claude Code" to any capable MCP client; nothing about the persona's five rules changes, only which model runs it.
- **What the server still does:** storage, the proposal gate (unchanged, ADR-0004), embeddings (`text-embedding-3-small`, `dimensions: 1024` — the one micro-API cost that must run server-side because it powers server-side retrieval indexes), and a **dormant** distill fallback. The in-app Distill button remains wired to `convex/ai/distill.ts`, but it now requires an explicit `OPENAI_API_KEY` on the live path and is never silently stubbed outside `AI_PROVIDER=stub`; without a configured key it honestly reports unavailable and points the user at their connected assistant (Task 2). It exists so the app is never *only* usable through a client connection — the manual/fallback path stays real, per the walking-skeleton principle (12 §Phase 1) — but it is no longer the primary generation path.
- **Auth, dual-mode:** bearer `atlas_sk_` keys are retained for header-capable agents (Codex, future agents); OAuth 2.1 + DCR + PKCE is added for connector-only clients like ChatGPT. Both resolve to the same `{userId, scopes}` model and the same tool registry (06 §1) — no client gets a different Atlas.
- **Cost model:** the user's ChatGPT Plus and Codex/Claude subscriptions absorb the conversational + drafting work at flat monthly rates; Atlas's own OpenAI metering drops to embeddings only (~pennies/month at personal-journal volume) plus the dormant distill path's occasional use. This replaces per-token metering of distill/connect/ask/reviews (ADR-0011's model table) as the steady-state cost for those tasks — ADR-0011's provider choice and model table are otherwise unchanged and still govern the dormant fallback and embeddings.

## Consequences
- 06 §1 gains OAuth 2.1 (AS metadata, DCR, PKCE, protected-resource metadata) alongside bearer keys; the "OAuth is post-MVP" line is deleted. 08 §6's accepted-risk row for bearer-only auth is resolved, not removed (record kept for history).
- 07 is generalized from "Claude app / Claude Code" to "ChatGPT connector, or Codex CLI / any MCP agent"; the persona doc (`docs/hermes-persona.md`) is extracted from 07 into its own self-contained file so it can be pasted directly into a ChatGPT project prompt — it's addressed to a GPT-based assistant, not assumed-Claude.
- Roadmap Phase 5 (MCP/Hermes) is pulled forward to now, as Phase M (12-roadmap), ahead of Phase 4 (knowledge depth) — the connector/OAuth work unlocks the subscription-cost model sooner and lets the owner start living the Hermes workflow while knowledge-depth features are still being built.
- Deferred by this pull-forward, re-slotted post-M: in-app search/Ask UI (retrieval lib ships now, UI later), the `ai/connect` pass, reviews prose (reviews ship as computed sections only; prose becomes on-demand via the connected client, not a server model call), the ops panel/issues/crashes surface, step-up auth for key creation, `atlas_list_reviews`/`atlas_get_review` MCP tools (added when reviews exist), and Apps-SDK interactive widgets for ChatGPT.
- Single-provider coupling (ADR-0011's consequence) now applies to a narrower surface — embeddings and the dormant fallback — while the bulk of "intelligence" cost and quality risk moves to whichever assistant the user already trusts and pays for. Atlas is no longer betting its usefulness on its own AI budget holding up under load.
